## 背景

- AI 写的方向全错，因为它没理解你真正想要什么
- 改了 20 个文件才发现思路不对，回滚成本巨大
- "已修复"三个字后面藏着没跑过测试的代码
- 上下文越堆越乱，最后连 AI 自己都搞混了

**OpenSpec 解决"想清楚再动手"的问题**——在 AI 写任何一行代码之前，先把需求、设计、任务拆清楚。

**Superpowers 解决"写完能信"的问题**——用 TDD、代码审查、验证铁则确保 AI 写出来的代码真的靠谱。

一个管"方向盘"，一个管"发动机"。合在一起，才是完整的 AI 工程化开发闭环。

## OpenSpec：规范驱动开发

> [Fission-AI/OpenSpec](https://github.com/Fission-AI/OpenSpec) — MIT 开源，支持 25+ AI 编程工具

### 核心理念

OpenSpec 的哲学很简单：**人类和 AI 在写代码之前先对齐需求**。

传统开发靠 PRD 文档、需求评审会、口头对齐——这些在 AI 编程时代全都失效了。你跟 AI 说"帮我做个暗黑模式"，它可能给你一个 toggle 按钮加 CSS 变量，也可能给你一套完整的设计系统。不写 Spec，AI 只能猜。

OpenSpec 用结构化的工件（Artifact）把"猜"变成"对齐"：

```
proposal.md  → 为什么做、做什么（意图与范围）
specs/       → 行为契约（Given/When/Then 场景）
design.md    → 怎么做（技术方案、架构决策）
tasks.md     → 分几步做（实现清单）
```

### 安装与初始化

```bash
# 安装 CLI（需要 Node.js ≥ 20.19.0）
npm install -g @fission-ai/openspec@latest

# 在项目中初始化
cd your-project
openspec init --tools claude
```

初始化后会生成 `.openspec` 目录和 `AGENTS.md` 文件。目录结构：

```
openspec/
├── specs/              # 系统当前行为规范（真相源）
│   └── <domain>/
│       └── spec.md
├── changes/            # 每个变更的工作区
│   └── <change-name>/
│       ├── proposal.md
│       ├── design.md
│       ├── tasks.md
│       └── specs/      # 增量规范（Delta Specs）
└── config.yaml
```

### 三步核心流程

**Step 1 — Propose（提议）**

```
/opsx:propose add-dark-mode
```

AI 根据你的需求描述，自动生成 `proposal.md`、`specs/`、`design.md`、`tasks.md` 四个工件。你审核、修改、确认。

**Step 2 — Apply（执行）**

```
/opsx:apply
```

AI 按照 `tasks.md` 的清单逐项实现，每完成一项就打勾。

**Step 3 — Archive（归档）**

```
/opsx:archive
```

变更归档到 `changes/archive/`，增量规范合并到主规范。知识不丢失，下一次变更站在上次的基础上。

### 三条可选路径

| 路径 | 适用场景 | 命令链 |
|------|---------|--------|
| **快速路径** | 需求明确的小功能 | `propose` → `apply` → `archive` |
| **探索路径** | 需求模糊、技术有歧义 | `explore` → `propose` → `apply` → `archive` |
| **精细路径** | 复杂项目、团队协作 | `new` → `continue`(逐个审核) → `apply` → `verify` → `archive` |

### Delta Specs：增量规范的精髓

OpenSpec 不是每次重写整个规范，而是用增量（Delta）描述变化：

```markdown
## ADDED Requirements
### 暗黑模式切换
- 用户 MUST 能够通过导航栏切换主题
- 系统 MUST 记住用户偏好到 localStorage

## MODIFIED Requirements
### 页面背景色
- 之前：固定白色
- 现在：跟随主题变量，暗黑模式下为 #1a1a2e
```

归档时，ADDED 追加到主规范，MODIFIED 替换旧条目，REMOVED 删除。规范像 Git 一样演进，有据可查。

## Superpowers：工程化开发工作流

> [obra/superpowers](https://github.com/obra/superpowers) ⭐ 39,700+ — Jesse Vincent 打造

### 核心理念

**Process over Prompt（流程大于提示词）**——给 AI 套上软件工程的纪律与护栏，让它像资深工程师一样思考。

Superpowers 不是一个技能，而是一套 **14 个子技能** 组成的完整 SDLC（软件开发生命周期）方法论。装上之后，AI 不再一上来就写代码，而是：

1. 先问你到底想要什么（brainstorming）
2. 拆分任务、制定计划（writing-plans）
3. 用 TDD 驱动实现（test-driven-development）
4. 每个任务完成后验证（verification-before-completion）
5. 代码审查（requesting-code-review）
6. 安全收尾（finishing-a-development-branch）

### 14 个子技能全景

| 类别 | 技能 | 用途 |
|------|------|------|
| **核心开发** | test-driven-development | 强制 TDD：先写失败测试，再写实现 |
| | systematic-debugging | 4 阶段调试：复现→定位→修复→验证 |
| | verification-before-completion | 没有验证证据就没有完成声明 |
| **计划设计** | brainstorming | 动手前必须先头脑风暴 |
| | writing-plans | 拆分为 2-5 分钟的细粒度任务 |
| | executing-plans | 批量执行（3 个任务一批）+ 人工检查点 |
| **代理统制** | subagent-driven-development | 每个任务派发新子代理 + 两阶段审查 |
| | dispatching-parallel-agents | 独立任务并行派发 |
| | using-superpowers | 元技能：始终检查是否有适用的技能 |
| **协作 Git** | requesting-code-review | 派发 code-reviewer 子代理 |
| | receiving-code-review | 技术性评估反馈，不是表演性同意 |
| | finishing-a-development-branch | 验证测试→4 选项→执行→清理 |
| | using-git-worktrees | 创建隔离工作区 + 基线验证 |
| | writing-skills | 用 TDD 方法写新技能 |

### 三条铁则（Iron Laws）

```text
🔴 TDD 铁则：没有测试就没有生产代码
🔴 调试铁则：没有根因分析就没有修复
🔴 验证铁则：没有验证证据就没有完成声明
```

这三条是不可协商的底线。AI 说"已修复"？拿出测试结果。AI 说"没问题"？贴出 diff 和命令输出。

### 安装

```bash
/plugin marketplace add obra/superpowers-marketplace
/plugin install superpowers@superpowers-marketplace
```

安装后无需手动触发——**技能自动生效**。AI 在执行任何任务前会自动检查是否有适用的子技能。

## 联合作战：OpenSpec × Superpowers 实战流程

单独用 OpenSpec，你有了"想清楚"的能力；单独用 Superpowers，你有了"写靠谱"的能力。但真正的威力在于它们的组合——**Spec 驱动规划，Superpowers 驱动执行**。

### 完整工作流

```text
┌─────────────────────────────────────────────────────────────┐
│                    Phase 1: 规划（OpenSpec 主导）             │
│                                                             │
│  /opsx:explore        ← 需求模糊时先探索                     │
│       ↓                                                     │
│  /opsx:propose        ← 生成 proposal + specs + design + tasks │
│       ↓                                                     │
│  人工审核工件           ← 确认意图、范围、验收条件              │
└─────────────────────────────────────────────────────────────┘
                          ↓
┌─────────────────────────────────────────────────────────────┐
│                Phase 2: 执行（Superpowers 主导）              │
│                                                             │
│  git worktree 隔离    ← using-git-worktrees                 │
│       ↓                                                     │
│  brainstorming        ← 对每个 task 做需求对齐               │
│       ↓                                                     │
│  TDD 实现             ← 先写失败测试 → 写实现 → 重构          │
│       ↓                                                     │
│  验证                  ← verification-before-completion      │
│       ↓                                                     │
│  代码审查              ← requesting-code-review              │
└─────────────────────────────────────────────────────────────┘
                          ↓
┌─────────────────────────────────────────────────────────────┐
│                Phase 3: 收尾（两者协同）                       │
│                                                             │
│  /opsx:verify         ← 检查实现是否匹配 Spec                │
│       ↓                                                     │
│  finishing-a-branch   ← 测试通过 → merge/PR/keep/discard     │
│       ↓                                                     │
│  /opsx:archive        ← 归档变更，增量规范合并                │
└─────────────────────────────────────────────────────────────┘
```

### 实战示例：实现订单导出功能

**Step 1：OpenSpec 规划**

```
/opsx:propose order-export
```

描述需求：

> 实现订单导出接口，支持按时间范围导出 CSV。单次最多 5000 条，时间范围不超过 31 天，只能导出当前租户数据。

OpenSpec 自动生成四个工件。你审核后确认：

```markdown
# proposal.md 要点
- 目标：订单导出 CSV 接口
- 范围内：CSV 生成、权限校验、数量限制、时间范围校验
- 范围外：异步导出、邮件通知、Excel 格式

# specs/ 关键场景
Given 用户是 tenant-A 的管理员
When  请求导出 tenant-A 在 2024-01-01 到 2024-01-31 的订单
Then  返回 CSV，字段为 order_no, amount, status, created_at

Given 用户尝试导出 tenant-B 的数据
When  请求导出
Then  返回 403，错误信息为 "无权访问该租户数据"

# tasks.md 清单
1.1 创建 OrderExportController
1.2 实现 OrderExportService（查询 + CSV 生成）
1.3 添加权限校验（只能导出当前租户）
1.4 添加数量限制（≤5000 条）
1.5 添加时间范围校验（≤31 天）
1.6 编写单元测试（覆盖 4 种边界场景）
```

**Step 2：Superpowers 执行**

进入执行阶段后，Superpowers 自动接管：

```text
🧠 brainstorming：对 task 1.1 做需求对齐
   → 确认 Controller 路径、请求参数格式、返回体结构

🌿 git worktree：创建隔离工作区
   → git worktree add ../project-order-export -b feat/order-export

🔴 TDD：先写失败测试
   → test: 导出正常路径返回 CSV
   → test: 超过 5000 条返回 400
   → test: 越权租户返回 403
   → test: 时间范围超限返回 400

🟢 实现：让测试通过
   → 实现 Controller、Service、权限校验

🔵 验证：verification-before-completion
   → 贴出 mvn test 输出
   → 贴出 git diff --stat

🔍 代码审查：requesting-code-review
   → 子代理检查权限绕过风险、SQL 注入、边界条件
```

**Step 3：协同收尾**

```
/opsx:verify    ← 检查实现是否覆盖了所有 Spec 场景
/opsx:archive   ← 归档，增量规范合并到主规范
```

Superpowers 的 `finishing-a-development-branch` 负责最终的测试验证和 Git 清理。

## 关键协同点

### 1. Spec 是 TDD 的输入源

Superpowers 的 TDD 要求"先写失败测试"——但测试用例从哪来？OpenSpec 的 `specs/` 目录就是天然的 BDD 场景库：

```markdown
# OpenSpec spec.md 中的场景
Given 订单数量超过 5000 条
When  请求导出
Then  返回 400，错误信息为 "单次导出不能超过 5000 条"
```

直接转化为测试用例：

```java
@Test
void shouldReturn400WhenExceedMaxRecords() {
    // Given
    when(orderRepository.countByDateRange(...)).thenReturn(5001L);
    // When
    var response = orderExportController.export(request);
    // Then
    assertEquals(400, response.getStatus());
    assertEquals("单次导出不能超过5000条", response.getMessage());
}
```

**Spec 定义"对不对"，TDD 确保"做对了"。**

### 2. verify 对齐 Spec 与实现

OpenSpec 的 `/opsx:verify` 检查三个维度：

| 维度 | 验证内容 |
|------|---------|
| **完整性** | 所有 task 是否完成？所有 requirement 是否实现？ |
| **正确性** | 实现是否匹配 Spec 意图？边界情况是否处理？ |
| **一致性** | 设计决策是否体现在代码结构中？命名是否一致？ |

这恰好弥补了 Superpowers 的一个盲区——Superpowers 的 `verification-before-completion` 关注的是"测试通过了没"，而 OpenSpec 的 `verify` 关注的是"实现跟 Spec 对上了没"。

### 3. archive 沉淀知识

Superpowers 的 `finishing-a-development-branch` 负责 Git 层面的收尾；OpenSpec 的 `archive` 负责知识层面的收尾。

归档后：
- 增量规范合并到主规范，下次开发站在新基础上
- 完整的变更记录（proposal → design → tasks → 实现）保留在 `archive/`
- AI 在后续变更中可以读取历史决策，避免重复讨论

## 高级技巧

### 1. 用 explore 消除技术歧义

需求模糊时别急着 propose。先用 `/opsx:explore` 让 AI 调研：

```
/opsx:explore
我想实现订单导出，但不确定：
1. CSV 生成用什么库？
2. 大数据量要不要流式写入？
3. 现有的权限框架怎么接入？
```

AI 会分析代码库，给出方案对比。调研结果直接喂给 `/opsx:propose`，Spec 质量大幅提升。

### 2. 用 parallel changes 处理紧急打断

正在开发"暗黑模式"，线上报了一个导出 Bug：

```
# 暂停当前工作
/opsx:new fix-export-null-pointer
/opsx:ff
/opsx:apply
/opsx:archive

# 恢复原工作
/opsx:apply add-dark-mode
```

每个变更独立管理，互不干扰。

### 3. 用 context 包喂给新会话

长任务超过两次纠正还不对？开新会话，带上结构化上下包：

```markdown
## 当前任务
实现订单导出接口（openspec/changes/order-export/）

## 必读文件
- openspec/changes/order-export/proposal.md
- openspec/changes/order-export/specs/
- openspec/changes/order-export/tasks.md
- src/main/java/.../OrderExportController.java

## 禁止修改
- 数据库已有字段名
- 全局异常格式
- 登录鉴权逻辑

## 验收命令
- mvn test
- mvn -Dtest=OrderExportServiceTest test
```

### 4. 贵模型规划，便宜模型执行

| 阶段 | 模型选择 | 任务 |
|------|---------|------|
| 规划 | Claude Opus 4.8 | `/opsx:propose`、审核工件、出方案 |
| 执行 | DeepSeek V4-Pro / GLM5.1 | `/opsx:apply`、按 task 编码、补测试 |
| 审查 | Claude Opus 4.8 | `/opsx:verify`、Review diff、最终验收 |

## 常见陷阱

### ❌ Spec 写成项目说明书

Spec 不是 README。只写行为契约（Given/When/Then），不写实现细节。判断标准：**如果实现方式变了但外部行为不变，就不该出现在 Spec 里。**

### ❌ 跳过 archive

不归档 = 知识碎片化。下次开发同类功能时，AI 读不到历史决策，又要重新讨论一遍。

### ❌ 同时改多个不相关的东西

一个变更只做一件事。"加暗黑模式 + 重构用户模块 + 修复导出 Bug"拆成三个变更。

### ❌ 不看 diff 直接信 AI 说"已修复"

```text
AI 说："已修复导出空指针问题"
你应该：git diff → 看改了什么 → mvn test → 看测试结果
```

### ❌ Spec 写完就不更新

Spec 是活文档。实现过程中发现 Spec 有误？回去改 Spec，不要在代码里绕过它。

## 总结

**OpenSpec 管"想清楚"，Superpowers 管"做对了"。**

- OpenSpec 的 Spec 是 Superpowers TDD 的输入源——没有 Spec，测试用例从哪来？
- Superpowers 的 TDD 铁则是 OpenSpec 的质量保障——有了 Spec 不测试，跟没写一样。
- OpenSpec 的 archive 沉淀知识，Superpowers 的 worktree 隔离风险。

两者组合，覆盖了 AI 编程的完整闭环：**需求对齐 → 方案设计 → 任务拆分 → TDD 实现 → 代码审查 → 验证归档**。

### 日常开发的节奏

把这些工具串起来，日常写需求时按这个节奏走：

1. **新建分支**，先确认工作区是干净的
2. **写一份轻量 Spec**（`/opsx:propose`），把目标、约束、验收标准说清楚
3. **看看有没有合适的 Skill**，比如 TDD、Code Review、前端设计、网页调研
4. **先让顶级模型出方案**，只讨论方案，不急着写代码
5. **方案确认后**，再让低价模型按 Task 一步步实现
6. **每完成一个 Task**，就跑测试、看 diff，然后小步提交
7. **当前 diff 稳住后**，再让顶级模型做一次 Review
8. **修掉 Review 里合理的问题**，再跑一遍测试
9. **合并前**，人工看关键 diff。涉及数据、权限、支付、定时任务这类改动时，再补一下文档、回滚方案或者灰度说明

这个流程比"一句话生成代码"慢一点。但慢的这点时间，通常会在后面赚回来——至少能少很多返工、回滚和线上排雷。

**短期原型可以大胆 Vibe**，先把东西跑起来再说；**但只要代码要长期维护**，还是得回到工程流程里。GitHub Flow 本身也是围绕分支、Pull Request、Review 和合并来组织协作，不是让人直接往主分支怼代码。

说白了，**AI 写代码越快，Git、测试、Review、Spec 这些老东西越不能丢。以前它们是为了约束人，现在还得顺手约束 AI。**

Vibe Coding 不是不行，但得有纪律。就像开车——你可以凭感觉开，但有了导航和安全带，你才能开得又快又稳。
