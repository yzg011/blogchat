# MYSQL
## 底层
### InnoDB
B+树
1. 非叶子节点只存储指针
2. 叶子节点存储数据、叶子节点之间通过指针相连，双向连链表结构
## 索引
### 什么是索引
- 索引（index）是帮助MySQL高效获取数据的数据结构（有序）
- 提高数据检索的效率，降低数据库的IO成本（不需要全表扫描）
- 通过索引列对数据进行排序，降低数据排序的成本，降低了CPU的消耗
### 创建索引的原则
- 数据量特别大，查询特别频繁（单表超过10w）
- 常用于查询、分组、排序的字段
- 区分多，重复少的字段
- 内容较多的字段，使用前缀索引`alter table x_test add index(x_name(7));`
- 使用联合索引，将查询中最常用作筛选条件的列放在最左边。
- 控制数量
- 用NOT NULL约束索引，不用空值

### 聚簇索引
- `数据和索引放到一块`，B+树的叶子节点保存了整行数据
- 只能有一个也必须要有，通常默认注解为聚簇索引，如果没主键则使用唯一索引
- 如果没主键没唯一，则自动生成一个rowid作为隐藏索引
- ![](./image/中间件-数据库.assets/中间件-数据库-20260102224909.png)
### 非聚簇索引
- 也称二级索引，`数据和索引分开存储`，B+树的叶子节点存放对应的主键
- ![](./image/中间件-数据库.assets/中间件-数据库-20260102225542.png)

### 联合索引
![](./image/中间件-数据库.assets/中间件-数据库-20260110122842.png)
**叶子节点**同样存储联合索引键值组合，但其指针部分不再指向下一级索引页，而是指向对应数据行所在的聚簇索引中的主键（RowID 或聚簇索引页地址）。因此，联合索引的叶子节点不直接包含完整的行数据，而是通过主键指针支持回表查询。
B+ 树遍历的规则是：
- 优先比较最左列（如 `name`）；
- 如果 `name` 相同，再继续比较 `age`；
- 若前两列都相同，最后比较 `sex`；
- 遍历过程从根节点逐层向下，直到叶子节点。
### 回表查询
- 先通过二级索引找到对应的主键值，
- 再通过找上面的主键索引进行聚簇索引查询，找到整行数据

### 覆盖索引
使用了索引，并且返回select所需的所有数据

### 索引失效
- 分索引列
	- 最左前缀，范围查询后面的索引会失效
	- 进行运算，推荐使用between
- 符号
	- `NOT IN`、`NOT EXISTS`、`<>`、`!=`
	- `OR`前后没有同时使用索引，但是可以用union all
	- `in`使用不当，断崖的in
	- `LIKE`前缀模糊索引失效，用覆盖索引可以走索引，后缀模糊都会走索引
- 数值
	- 隐式类型转换、字符集不一致
- 其他
	- 低选择性和数据量过小索引

### 主键索引和唯一索引区别

| 特性          | 主键索引（Primary Key Index） | 唯一索引（Unique Index） |
| ----------- | ----------------------- | ------------------ |
| 唯一性         | 强制唯一且不能包含 `NULL` 值      | 强制唯一，通常允许 `NULL` 值 |
| 允许 `NULL` 值 | 不允许                     | 通常允许（具体行为视数据库系统而定） |
| 数量          | 每个表只能有一个主键索引            | 每个表可以有多个唯一索引       |
| 存储顺序        | 通常是聚簇索引，数据行按照主键顺序存储     | 通常是非聚簇索引，数据行存储顺序不变 |


## 事务
### 事务特性（ACID）
- 原子性：全部成功或全部失败；undo log⽇志保证
- 一致性：完成时候，状态一致
- 隔离性：并发操作下独立环境不受影；由MVCC来保证
- 持久性：事务提交后，落盘数据永久；redo log来保证

### 事务问题
- 脏读：一个事务读到另一个事务还没提交的数据
- 不可重复读：同一个事务先后读取同一条记录，两次读取的数据不同
- 幻读：一个事务查询没有查到，但是插入时候发现已经存在

### 隔离级别
- 读未提交：允许读取尚未提交的数据变更
- 读已提交：允许读取并发事务已经提交的数据，oracle默认
- 可重复读：对同一字段的多次读取结果都是一致的，mysql默认
- 串行：事务完成后再接着下一个事务

### MVCC
- 多版本并发控制
	- 隐藏字段：事务id+回滚指针，指向上一个事务版本地址
	- undo log：回滚日志，存储老版本日志，版本链，多事务并行操作某一行记录，记录不同事务修改数据的版本，通过roll_pointer指针形成链表
	- readView：解决一个事务版本选择的问题，根据当前一些事务ID和匹配规则判断需要访问哪个版本的数据
- 不同事务或者相同事务对同一条记录进行修改，会导致该记录的undolog生成一条记录版本链表，链表头是最新的旧纪录，链表尾部是最早的旧纪录

## 锁机制
### 行级锁
- **共享锁**：又称读锁，只有当所有 读 锁都释放后，才能加 写 锁。
- **排他锁**：又称写锁，事务对某一行加上 写 锁后，其他事务既不能对该行加  读 锁，也不能加 写 锁，只能等待该 写 锁释放。
- InnoDB 的行级锁是通过**索引**来实现的，如果查询语句没有使用索引，那么 InnoDB 会使用表级锁，锁住整个表。
### 表级锁
- **表共享读锁**：多个事务可以同时获取表的读锁，此时只能读取表中的数据，不能修改。
- **表独占写锁**：一个事务获取表的写锁后，只有该事务可以对表进行读写操作，其他事务既不能读也不能写，必须等待写锁释放。
- **意向锁**：
- **自增锁**：当事务插入包含自增列的数据时，会获取自增锁，生成自增的值
### 特殊的锁
- **间隙锁（Gap Lock）**：锁定索引记录之间的“间隙”，防止其他事务在这个间隙中插入新记录，从而**解决“幻读”问题**（在 `REPEATABLE READ` 隔离级别下生效）。
	- 例如，`SELECT * FROM users WHERE id BETWEEN 10 AND 20 FOR UPDATE;` 不仅会锁住 `id=10, 11, ... ,20` 的记录，还会锁住 `(10, 11), (11,12), ..., (19,20)` 以及 `(20, +∞)` 这些间隙。

- **临键锁（Next-Key Lock）**：**行级锁与间隙锁的组合**，锁定一条记录及其前面的间隙。它是 InnoDB 在 `REPEATABLE READ` 隔离级别下**默认的行锁算法**。
	- 例如，假设索引有值 10, 11, 13，那么 `Next-Key Lock` 锁定的范围可能是 `(-∞, 10], (10, 11], (11, 13], (13, +∞)`。

## 三大日志
MySQL InnoDB 引擎使用 **redo log(重做日志)** 保证事务的**持久性**，使用 **undo log(回滚日志)** 来保证事务的**原子性**。
`MySQL`数据库的**数据备份、主备、主主、主从**都离不开`binlog`，需要依靠`binlog`来同步数据，保证数据一致性。
### redo log
物理日志
- 专门记录物理更改，用于保证数据的持久性和崩溃恢复。
- 
![|595x323](./image/中间件-数据库.assets/中间件-数据库-20260103160955.png)

### bin log
逻辑日志
- 记录逻辑更改，用于主从复制和数据恢复，记录逻辑操作。
- Binlog的写入通常在Redo Log之后，以确保数据的一致性。
![](./image/中间件-数据库.assets/中间件-数据库-20260103182744.png)
### undo log
逻辑日志
- 记录数据改变前的状态，用于事务的回滚和多版本并发控制（MVCC）。
- 提供回滚
	如果执行过程中遇到异常的话，我们直接利用 **回滚日志** 中的信息将数据回滚到修改之前的样子即可！并且，回滚日志会先于数据持久化到磁盘上。
	当执行一条操作时，日志会记录相反的记录




## 调优
### 定位慢查询
一般写完SQL后都会看看执行计划，看看type、key和row是否走了索引
如果是线上的用的Prometheus和Arthas进行查看哪些执行时间过长


### 优化
#### 建表开始
- 选择合适的字段类型、数值长度，(tinyint,int,bigint)
- 必要的通用主键
- 字段不宜超过20个
- 将字段定义为 NOT NULL
- 评估需要加索引
- 不需要严格遵循3NF
- 统一的字符集
- 枚举类型注释要说清楚
- 考虑是否需要分库分表
- 大字段考虑mongdb

#### SQL语句优化
- 使用select指明字段，避免使用select*
- 避免索引失效写法
- union all代替union，把过滤交给业务层
- where避免使用表达式操作
- 查存在的用EXISTS代替in
- 小表驱动大表
- 写完后最好执行一下执行计划，看看哪里查询的时间长，可以给服务层处理，然后对症下药
#### 索引优化
创建原则+索引失效
#### 主从复制、读写分离、分库分表
使用seata搭建主从复制
主从复制
- 主库把变化的数据写入到binlog日志中
- 从库读取主库的binlog日志，写入到从库的中继日志Relay log
- 从库重做中继日志的事件，将改变反应它自己的数据
分库分表
- 垂直分库：将不同的业务拆分到不同库中
- 水平分库：将一个库的数据按特定的路由规则进行拆分到多个库中
- 垂直分表：将不常用或者大字段拆分到不同表中，
- 水平分表：将一个表的数据按特定的路由规则进行拆分到多个表中（同一个库内）
分库要注意
- 分布式事务一致性问题
- 跨界点关联查询
- 跨节点分页、排序函数
- 主键避重
