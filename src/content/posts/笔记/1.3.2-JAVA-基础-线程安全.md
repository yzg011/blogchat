# Java 线程安全指南

## 一、⚡ 速查表

### 1.1 解决方案对比总表

| 方案 | 适用问题 | 性能 | 使用复杂度 | 适用场景 |
|------|----------|------|------------|----------|
| `AtomicInteger` | 单个数值的原子操作 | 极高（无锁）| 低 | 计数器、序列号 |
| `LongAdder` | 高并发累加 | 最高 | 低 | 统计、监控 |
| `AtomicReference` | 单个对象引用原子替换 | 高 | 低 | 配置热更新 |
| `AtomicStampedReference` | 带版本号的原子引用 | 高 | 中 | 防 ABA 问题 |
| `ConcurrentHashMap` | 并发 Map | 高 | 低 | 缓存、计数 Map |
| `CopyOnWriteArrayList` | 读多写少的 List | 读极高/写低 | 低 | 监听器、配置 |
| `BlockingQueue` | 线程间安全传递 | 高 | 低 | 生产者-消费者 |
| `ThreadLocal` | 线程间隔离 | 极高（无竞争）| 低（注意泄漏）| 会话、连接 |
| `不可变对象` | 防止修改 | 极高 | 中（需要设计）| 值对象、DTO |
| `synchronized` | 通用互斥 | 中 | 低 | 简单临界区 |
| `ReentrantLock` | 高级互斥 | 中高 | 中 | 需要超时/公平 |
| `ReadWriteLock` | 读多写少 | 读高/写中 | 中 | 缓存、配置 |
| `StampedLock` | 极致性能读写分离 | 最高 | 高 | 高性能读多写少 |
| `CountDownLatch` | 等待 N 事件完成 | 高 | 低 | 初始化、聚合 |
| `CyclicBarrier` | 多线程分阶段同步 | 高 | 低 | 并行计算 |
| `Semaphore` | 控制并发数 | 高 | 低 | 限流、连接池 |

### 1.2 选择决策树

```
需要解决什么问题？
│
├── 单个变量的并发修改？
│   ├── 整型/长整型/布尔 → AtomicInteger / AtomicLong / AtomicBoolean
│   ├── 高并发纯累加   → LongAdder（比 AtomicLong 更快）
│   └── 对象引用        → AtomicReference（有ABA问题则用 AtomicStampedReference）
│
├── 共享集合的并发访问？
│   ├── Map → ConcurrentHashMap
│   ├── List（读多写少）→ CopyOnWriteArrayList
│   ├── List（写多）→ Collections.synchronizedList 或加锁
│   ├── Queue（线程间传数据）→ LinkedBlockingQueue / ArrayBlockingQueue
│   └── Set → CopyOnWriteArraySet / ConcurrentSkipListSet
│
├── 根本不想共享？
│   ├── 每个线程独立数据 → ThreadLocal
│   ├── 数据只读         → 不可变对象（final + 防御性复制）
│   └── 数据只在方法内   → 局部变量（天然安全）
│
├── 多个操作必须原子执行？
│   ├── 简单场景 → synchronized
│   ├── 需要超时/可中断 → ReentrantLock
│   ├── 读多写少     → ReadWriteLock（或 StampedLock 追求极致性能）
│   └── 多个变量同时修改 → 锁住包含这些变量的对象
│
└── 线程之间需要协作？
    ├── 主线程等 N 个子线程完成 → CountDownLatch
    ├── N 个线程互相等，一起进入下一阶段 → CyclicBarrier
    ├── 限制同时执行的线程数 → Semaphore
    └── 两线程互换数据 → Exchanger
```

### 1.4 三个黄金原则

```
原则1：优先不共享（最安全）
  → 局部变量 > 不可变对象 > ThreadLocal > 共享+同步

原则2：共享时，用最合适的工具（最小代价）
  → 单变量   → 原子类（无锁，性能最高）
  → 集合     → 并发集合（按场景选型）
  → 复杂操作 → 锁（先 synchronized，再 ReentrantLock）

原则3：每个并发决策都要考虑后果（最负责任）
  → 有没有 finally unlock/release/remove？
  → 有没有设置超时？
  → 异常路径有没有清理资源？
```

## 二、先搞懂：什么是线程安全问题？

### 2.1 用最简单的例子理解

```java
// 模拟两个线程同时给同一个账户加钱
public class BankAccount {
    private int balance = 1000; // 初始余额 1000 元

    public void deposit(int amount) {
        balance = balance + amount; // 看起来很简单，但这行代码不是原子的！
    }
}
```

**问题所在**：`balance = balance + amount` 实际上分三步执行：

```
步骤1：从内存读取 balance 的值（读）
步骤2：执行加法计算         （算）
步骤3：将结果写回内存       （写）
```

两个线程同时执行时，可能发生这种情况：

```
时间线：
线程A：读取 balance=1000
线程B：读取 balance=1000   ← 线程B也读到了 1000，因为A还没写回
线程A：计算 1000+500=1500
线程B：计算 1000+300=1300
线程A：写回 balance=1500
线程B：写回 balance=1300   ← 覆盖了线程A的结果！

预期结果：1000 + 500 + 300 = 1800
实际结果：1300  ← 500元凭空消失了！
```

这就是**竞态条件（Race Condition）**，是线程安全问题最典型的表现。

### 2.2 线程安全问题的三大根源

```
根源1：可见性问题
  线程A修改了变量，线程B看不到最新值（因为CPU缓存）
  → 用 volatile 或 synchronized 解决

根源2：原子性问题
  一个"逻辑操作"实际上由多个CPU指令组成，中间可能被打断
  → 用原子类（Atomic）或锁解决

根源3：有序性问题
  JVM/CPU 为了优化会重排指令顺序，在多线程下可能出问题
  → 用 volatile 或 happens-before 原则解决
```

### 2.3 解决方案全景图

```
Java 线程安全解决方案
├── 原子类（Atomic*）         → 单个变量的无锁原子操作
├── 线程安全集合               → ConcurrentHashMap 等
├── 避免共享                  → 不共享就没有竞争
│   ├── ThreadLocal           → 每个线程独立副本
│   ├── 局部变量              → 栈上数据，天然线程安全
│   └── 不可变对象（final）   → 不能改就不会有问题
├── 锁机制                    → synchronized / ReentrantLock
├── 并发工具类                → CountDownLatch / Semaphore 等
└── volatile                  → 轻量级可见性保证
```

### 3.1 为什么普通集合不是线程安全的？

```java
// ArrayList 内部是数组，add 操作分多步：

// 3. size++

HashMap<String, Integer> map = new HashMap<>();
```

### 3.3 ConcurrentHashMap

> **输入**: `put(K key, V value)` → 返回旧值或 null
> **输出**: `get(K key)` → 返回 V 或 null
> **作用**: 高并发线程安全的 HashMap，内部分段锁/红黑树，读写性能极高
> **对比**: HashMap（线程不安全） / Hashtable（全表锁，慢） / Collections.synchronizedMap（全表锁，慢）

```java
import java.util.concurrent.ConcurrentHashMap;

public class ConcurrentHashMapDemo {

    ConcurrentHashMap<String, Integer> map = new ConcurrentHashMap<>();

    // ===== 基本用法（和 HashMap 一样）=====
    public void basicUsage() {
        map.put("key", 1);
        Integer value = map.get("key");
        map.remove("key");
        map.containsKey("key");
    }

    // ===== 重要：这些操作是原子的（HashMap 没有）=====
    public void atomicOperations() {

        // putIfAbsent：key 不存在时才 put（原子操作）
        // 返回旧值（如果key已存在），或 null（如果是新插入的）
        Integer old = map.putIfAbsent("counter", 0);

        // computeIfAbsent：key 不存在时，用函数计算并放入（原子）
        // 常用于缓存初始化
        map.computeIfAbsent("list", k -> 100); // 如果 "list" 不存在，初始化为100

        // compute：根据现有值计算新值（原子）
        // 等价于"先读后写"的原子版本
        map.compute("counter", (k, v) -> v == null ? 1 : v + 1); // 原子累加

        // merge：合并操作（原子）
        // 如果 key 不存在，直接放入 value；如果存在，用 remappingFunction 合并
        map.merge("count", 1, Integer::sum); // 原子地 count += 1
    }

    // ===== 实际用例：统计单词出现次数 =====
    public void wordCount(String[] words) {
        for (String word : words) {
            // ✅ 原子累加，线程安全
            map.merge(word, 1, Integer::sum);

            // 等价写法（更明确）：
            // map.compute(word, (k, v) -> v == null ? 1 : v + 1);
        }
    }

    // ===== 批量操作（JDK 8+，并行执行）=====
    public void bulkOperations() {
        // forEach：并行遍历（parallelismThreshold=1 表示总是并行）
        map.forEach(1, (k, v) -> System.out.println(k + "=" + v));

        // search：并行查找，返回第一个非 null 的结果
        String result = map.search(1, (k, v) -> v > 10 ? k : null);

        // reduce：并行归约
        long total = map.reduceValues(1, v -> (long) v, Long::sum);
    }
}
```

### 3.4 CopyOnWriteArrayList

> **输入**: `add(E e)`, `addAll(Collection)`
> **输出**: `get(int index)` → E, `iterator()` → 快照遍历
> **作用**: 读操作完全无锁，写操作复制整个数组，适合读远多于写的场景
> **对比**: ArrayList（线程不安全） / Collections.synchronizedList（读写都加锁） / Vector（全表锁）

```java
import java.util.concurrent.CopyOnWriteArrayList;

/**
 * 核心原理：写时复制
 * - 读操作：直接读，完全无锁
 * - 写操作：复制整个数组，在副本上修改，然后原子替换引用
 *
 * 适用场景：读远多于写（如：监听器列表、配置列表、路由规则）
 * 不适用：写操作频繁（每次写都复制整个数组，内存和 CPU 开销大）
 */
public class CopyOnWriteDemo {

    // 场景：事件监听器列表（注册一次，大量事件触发时遍历）
    private final CopyOnWriteArrayList<EventListener> listeners = new CopyOnWriteArrayList<>();

    // 写操作：添加监听器（少量，触发复制）
    public void addListener(EventListener listener) {
        listeners.add(listener);  // 内部：复制数组，添加，替换引用
    }

    // 读操作：触发事件（高频，无锁）
    public void fireEvent(Event event) {
        // 遍历时完全无锁，即使同时有 add 操作也不影响当前遍历
        for (EventListener listener : listeners) {
            listener.onEvent(event);
        }
    }

    // 批量添加比逐个添加效率高（只复制一次数组）
    public void addAllListeners(List<EventListener> newListeners) {
        listeners.addAll(newListeners);
    }
}
```

### 3.5 阻塞队列

> **输入**: `put(E e)`（满时阻塞）, `offer(E e, timeout)`（超时返回 false）, `add(E e)`（满时抛异常）
> **输出**: `take()`（空时阻塞）, `poll(timeout)`（超时返回 null）
> **作用**: 生产者-消费者模式的核心工具，天然实现流量控制和背压
> **对比**: 
> - ArrayBlockingQueue：容量固定，数组实现，性能稳定，单锁
> - LinkedBlockingQueue：可选有界，链表实现，双锁吞吐更高，无界有 OOM 风险
> - ConcurrentLinkedQueue：无锁队列，非阻塞，无界

```java
import java.util.concurrent.ArrayBlockingQueue;
import java.util.concurrent.BlockingQueue;
import java.util.concurrent.LinkedBlockingQueue;

/**
 * 阻塞队列是实现"生产者-消费者"模式的最佳工具
 *
 * 核心特性：
 * - 队列满时，put() 自动阻塞生产者（等待空间）
 * - 队列空时，take() 自动阻塞消费者（等待数据）
 * - 天然实现了背压和流量控制
 */
public class BlockingQueueDemo {

    // 有界队列（推荐生产使用，防止内存无限增长）
    BlockingQueue<String> queue = new ArrayBlockingQueue<>(100);

    // ===== 生产者线程 =====
    class Producer implements Runnable {
        @Override
        public void run() {
            try {
                for (int i = 0; i < 1000; i++) {
                    String task = "task-" + i;

                    // put()：队列满时阻塞（推荐，保证数据不丢失）
                    queue.put(task);
                    System.out.println("生产: " + task);

                    // offer(task, 1, TimeUnit.SECONDS)：等1秒，超时返回false
                    // boolean success = queue.offer(task, 1, TimeUnit.SECONDS);

                    // add(task)：队列满时抛异常（不推荐）
                }
            } catch (InterruptedException e) {
                Thread.currentThread().interrupt();
            }
        }
    }

    // ===== 消费者线程 =====
    class Consumer implements Runnable {
        @Override
        public void run() {
            try {
                while (true) {
                    // take()：队列空时阻塞（推荐）
                    String task = queue.take();
                    System.out.println("消费: " + task);
                    process(task);

                    // poll(1, TimeUnit.SECONDS)：等1秒，超时返回null
                    // String task = queue.poll(1, TimeUnit.SECONDS);
                    // if (task == null) break; // 超时退出
                }
            } catch (InterruptedException e) {
                Thread.currentThread().interrupt();
            }
        }

        void process(String task) { /* 业务处理 */ }
    }

    // ===== 启动生产者消费者 =====
    public void start() {
        Thread producer  = new Thread(new Producer());
        Thread consumer1 = new Thread(new Consumer()); // 可以有多个消费者
        Thread consumer2 = new Thread(new Consumer());

        producer.start();
        consumer1.start();
        consumer2.start();
    }
}
```

## 四、原子类（无锁高性能）

### 4.1 什么是原子类？

原子类是 Java 提供的一组专门用于线程安全操作的类，位于 `java.util.concurrent.atomic` 包下。

**核心原理**：基于 **CAS（Compare And Swap，比较并交换）** 指令，是 CPU 级别的原子操作，不需要加锁，性能远高于 `synchronized`。

**CAS 原理通俗解释**：

```
CAS(内存地址, 预期旧值, 新值)

操作语义：
"如果内存里的值还是我之前读到的值（没被别人改过），就把它改成新值；
 否则说明被人改了，我放弃这次修改，重新来过"

这就是"乐观锁"思想：先假设没有冲突，冲突了就重试
```

### 4.2 原子类全家族

```
java.util.concurrent.atomic 包
│
├── 基本类型原子类
│   ├── AtomicInteger      整型原子类
│   ├── AtomicLong         长整型原子类
│   └── AtomicBoolean      布尔型原子类
```

### 4.3 AtomicInteger：基础用法完全示例

```java
import java.util.concurrent.atomic.AtomicInteger;

public class AtomicIntegerDemo {

    // 声明原子整型（替代 int/Integer）
    private AtomicInteger count = new AtomicInteger(0);

    // ===== 基本读写 =====
    public int get() {
        return count.get();             // 获取当前值
    }

    public void set(int value) {
        count.set(value);               // 设置新值（原子）
    }

    // ===== 原子增减 =====
    public int increment() {
        return count.incrementAndGet(); // ++count，返回新值
    }

    public int decrement() {
        return count.decrementAndGet(); // --count，返回新值
    }

    public int addFive() {
        return count.addAndGet(5);      // count += 5，返回新值
    }

    public int getAndIncrement() {
        return count.getAndIncrement(); // count++，返回旧值（注意：返回的是加之前的值！）
    }

    // ===== CAS 操作（最强大的方法）=====
    public boolean casUpdate(int expect, int update) {
        // 只有当前值等于 expect 时，才将值改为 update
        return count.compareAndSet(expect, update);
    }

    // ===== 函数式更新（JDK 8+）=====
    public int updateWithFunction() {
        // updateAndGet：用函数计算新值，原子更新
        return count.updateAndGet(current -> current * 2); // 原子地乘以2

        // 或者 accumulateAndGet：结合当前值和外部值
        // return count.accumulateAndGet(10, Integer::max); // 原子地取 max(current, 10)
    }

    // ===== 实际应用：线程安全计数器 =====
    public static void main(String[] args) throws InterruptedException {
        AtomicInteger safeCounter = new AtomicInteger(0);

        // 创建 1000 个线程同时执行 +1
        Thread[] threads = new Thread[1000];
        for (int i = 0; i < 1000; i++) {
            threads[i] = new Thread(() -> safeCounter.incrementAndGet());
            threads[i].start();
        }

        // 等所有线程完成
        for (Thread t : threads) t.join();

        System.out.println("最终结果：" + safeCounter.get()); // 一定是 1000，永远不会少
    }
```

### 4.4 AtomicLong vs LongAdder：高并发下的选择

> **AtomicLong**：适合读多写少，需要精确当前值。内部所有线程竞争同一个变量。
> **LongAdder**：适合写多读少（统计、计数），高并发吞吐更高。内部维护多个 Cell 减少竞争，读取时求和，牺牲精度换吞吐。

```java
import java.util.concurrent.atomic.AtomicLong;
import java.util.concurrent.atomic.LongAdder;

public class CounterComparison {

    AtomicLong atomicCounter = new AtomicLong(0);
    LongAdder  adderCounter  = new LongAdder();

    // 场景1：统计接口调用次数（写多读少）→ 用 LongAdder
    public void recordApiCall() {
        adderCounter.increment(); // 高并发下性能更好
    }
    public long getApiCallCount() {
        return adderCounter.sum(); // 读取时求和（不是瞬时精确值）
    }

    // 场景2：实现序列号生成（需要精确当前值）→ 用 AtomicLong
    public long generateSerialNo() {
        return atomicCounter.incrementAndGet(); // 保证每次返回唯一且递增的值
    }
}
```

### 4.6 AtomicStampedReference：解决 ABA 问题

```java
import java.util.concurrent.atomic.AtomicStampedReference;

/**
 * ABA 问题说明：
 * 线程A 读取值为 "A"
 * 线程B 把值改成 "B"
 * 线程C 又把值改回 "A"
 * 线程A 做 CAS，发现值还是 "A"，以为没有变化，CAS 成功
 * → 但实际上值已经被改过两次了！在某些业务场景下这是错误的（如链表操作）
 *
 * 解决方案：加版本号（stamp），每次修改版本号+1
 */
public class ABADemo {

    // 带版本号的原子引用
    // 第二个参数是初始 stamp（版本号），从 1 开始
    AtomicStampedReference<String> ref =
        new AtomicStampedReference<>("A", 1);

    public void update() {
        // 读取当前值和版本号
        int[]  stampHolder = new int[1];
        String value       = ref.get(stampHolder); // 获取值，版本号存入数组
        int    stamp       = stampHolder[0];

        System.out.println("当前值: " + value + ", 版本号: " + stamp);

        // CAS 更新：必须值和版本号都匹配才能成功
        boolean success = ref.compareAndSet(
            value,       // 预期旧值
            "B",         // 新值
            stamp,       // 预期旧版本号
            stamp + 1    // 新版本号
        );
        System.out.println("更新成功: " + success);
    }
}
```

### 6.1 CountDownLatch：等待 N 件事完成

```java
import java.util.concurrent.CountDownLatch;

/**
 * CountDownLatch：一个线程等待其他 N 个线程完成某件事
 * 就像发令枪：等所有运动员到起跑线，一声令下同时起跑
 * 或：等所有子任务完成，主线程再汇总结果
 *
 * 特点：只能用一次！count 减到 0 后不能重置（要重置用 CyclicBarrier）
 */
public class CountDownLatchDemo {

    // 场景1：等待所有子线程完成后汇总
    public void waitForAllTasks() throws InterruptedException {
        int taskCount = 5;
        CountDownLatch latch = new CountDownLatch(taskCount); // 初始计数5

        List<String> results = Collections.synchronizedList(new ArrayList<>());

        for (int i = 0; i < taskCount; i++) {
            int taskId = i;
            new Thread(() -> {
                try {
                    String result = executeTask(taskId);
                    results.add(result);
                } finally {
                    latch.countDown(); // 每个线程完成后 count-1
                }
            }).start();
        }

        latch.await(); // 主线程在这里等待，直到 count 变为 0
        // 或带超时：latch.await(10, TimeUnit.SECONDS);
        System.out.println("所有任务完成，结果: " + results);
    }

    // 场景2：模拟并发起跑（所有线程同时开始）
    public void simulateConcurrentStart() throws InterruptedException {
        CountDownLatch startGun = new CountDownLatch(1); // 发令枪，只需倒数1次
        CountDownLatch allDone  = new CountDownLatch(10); // 等10个线程完成

        for (int i = 0; i < 10; i++) {
            int id = i;
            new Thread(() -> {
                try {
                    startGun.await(); // 所有线程在这里等待发令
                    System.out.println("线程" + id + " 开始执行");
                } catch (InterruptedException e) {
                    Thread.currentThread().interrupt();
                } finally {
                    allDone.countDown();
                }
            }).start();
        }

        Thread.sleep(100); // 确保所有线程已就绪
        startGun.countDown(); // 发令！所有线程同时开始

        allDone.await(); // 等所有线程结束
        System.out.println("全部完成");
    }

    private String executeTask(int id) {
        return "result-" + id;
    }
}
```

### 6.2 CyclicBarrier：多线程相互等待（可重用）

```java
import java.util.concurrent.CyclicBarrier;

/**
 * CyclicBarrier：所有线程都到达屏障点后，大家一起继续执行
 * 可以循环使用（这是和 CountDownLatch 的核心区别）
 *
 * 场景：分阶段计算
 * 例如：所有线程完成第1阶段 → 汇总 → 所有线程再做第2阶段
 */
public class CyclicBarrierDemo {

    public void phaseCalculation() {
        int threadCount = 4;

        // 当所有线程到达屏障时，执行 barrierAction（在最后到达的那个线程中执行）
        CyclicBarrier barrier = new CyclicBarrier(threadCount, () -> {
            System.out.println("=== 所有线程完成当前阶段，开始下一阶段 ===");
        });

        for (int i = 0; i < threadCount; i++) {
            int id = i;
            new Thread(() -> {
                try {
                    // 第一阶段
                    System.out.println("线程" + id + " 完成阶段1");
                    barrier.await(); // 等所有线程完成阶段1

                    // 第二阶段（所有线程同时开始）
                    System.out.println("线程" + id + " 完成阶段2");
                    barrier.await(); // CyclicBarrier 可以重复使用！

                    // 第三阶段
                    System.out.println("线程" + id + " 完成阶段3");
                    barrier.await();

                } catch (Exception e) {
                    Thread.currentThread().interrupt();
                }
            }).start();
        }
    }
}
```

## 七、锁机制详解与对比

### 7.1 synchronized：最基础的锁

```java
public class SynchronizedDemo {

    private int count = 0;
    private final Object lock = new Object(); // 锁对象

    // 方法级锁（锁的是 this 对象）
    public synchronized void incrementMethod() {
        count++;
    }
```

### 7.2 ReentrantLock：功能强大的可重入锁

```java
import java.util.concurrent.locks.ReentrantLock;

public class ReentrantLockDemo {

    private final ReentrantLock lock = new ReentrantLock();
    // 公平锁：按等待顺序分配，防止饥饿，但吞吐量稍低
    private final ReentrantLock fairLock = new ReentrantLock(true);

    private int count = 0;

    // 基本用法
    public void basicUsage() {
        lock.lock(); // 获取锁（阻塞等待）
        try {
            count++;
        } finally {
            lock.unlock(); // 必须在 finally 中释放！
        }
    }

    // tryLock：尝试获取锁，不阻塞
    public boolean tryUpdate() {
        if (lock.tryLock()) { // 立即返回 true 或 false
            try {
                count++;
                return true;
            } finally {
                lock.unlock();
            }
        }
        return false; // 锁被占用，直接返回 false，不等待
    }

    // tryLock(timeout)：等待指定时间
    public boolean tryUpdateWithTimeout() throws InterruptedException {
        if (lock.tryLock(3, TimeUnit.SECONDS)) { // 最多等3秒
            try {
                count++;
                return true;
            } finally {
                lock.unlock();
            }
        }
        return false; // 3秒内没拿到锁
    }

    // 可中断的锁等待
    public void interruptibleLock() throws InterruptedException {
        lock.lockInterruptibly(); // 等待锁时，可以被 interrupt() 中断
        try {
            count++;
        } finally {
            lock.unlock();
        }
    }
}
```

### 7.3 ReadWriteLock：读写分离，提高并发

```java
import java.util.concurrent.locks.ReadWriteLock;
import java.util.concurrent.locks.ReentrantReadWriteLock;

/**
 * 读写锁：读-读 不互斥（多线程同时读），读-写 互斥，写-写 互斥
 * 适用场景：读多写少（缓存、配置、路由表）
 */
public class ReadWriteLockDemo {

    private final ReadWriteLock rwLock  = new ReentrantReadWriteLock();
    private final Map<String, String>  cache  = new HashMap<>();

    // 读操作：多个线程可以同时读
    public String get(String key) {
        rwLock.readLock().lock();
        try {
            return cache.get(key); // 并发读，不互斥
        } finally {
            rwLock.readLock().unlock();
        }
    }

    // 写操作：只有一个线程能写，写时阻塞所有读
    public void put(String key, String value) {
        rwLock.writeLock().lock();
        try {
            cache.put(key, value);
        } finally {
            rwLock.writeLock().unlock();
        }
    }

    // 锁升级（读锁 → 写锁）：ReentrantReadWriteLock 不支持！
    public void upgradeWrong(String key, String value) {
        rwLock.readLock().lock();
        try {
            if (!cache.containsKey(key)) {
                // ❌ 持有读锁时不能直接获取写锁，会死锁！
                rwLock.writeLock().lock(); // 会在这里永远阻塞
                try { cache.put(key, value); }
                finally { rwLock.writeLock().unlock(); }
            }
        } finally {
            rwLock.readLock().unlock();
        }
    }

    // ✅ 正确的升级方式：先释放读锁，再获取写锁
    public void upgradeCorrect(String key, String value) {
        rwLock.readLock().lock();
        boolean needWrite;
        try {
            needWrite = !cache.containsKey(key);
        } finally {
            rwLock.readLock().unlock(); // 先释放读锁
        }

        if (needWrite) {
            rwLock.writeLock().lock(); // 再获取写锁
            try {
                if (!cache.containsKey(key)) { // 再次检查（防止双重检查问题）
                    cache.put(key, value);
                }
            } finally {
                rwLock.writeLock().unlock();
            }
        }
    }
}
```

### 7.4 StampedLock：JDK 8+ 最高性能读写锁

```java
import java.util.concurrent.locks.StampedLock;

/**
 * StampedLock：比 ReadWriteLock 更高性能
 * 核心特性：乐观读（Optimistic Reading）
 * - 乐观读：不加锁直接读，读完后验证是否被写过
 * - 如果没被写过：最优路径，完全无锁
 * - 如果被写过：降级为悲观读锁重试
 */
public class StampedLockDemo {

    private final StampedLock stampedLock = new StampedLock();
    private double x = 0, y = 0;

    // 写操作
    public void move(double deltaX, double deltaY) {
        long stamp = stampedLock.writeLock(); // 获取写锁，返回 stamp（版本号）
        try {
            x += deltaX;
            y += deltaY;
        } finally {
            stampedLock.unlockWrite(stamp); // 必须用写锁的 stamp 释放
        }
    }

    // 乐观读（最高性能的读取方式）
    public double distanceFromOrigin() {
        long stamp = stampedLock.tryOptimisticRead(); // 获取乐观读 stamp（不加锁）
        double curX = x, curY = y;                    // 读取数据

        if (!stampedLock.validate(stamp)) {           // 检查：读取期间有写操作吗？
            // 有写操作，乐观读失败，升级为悲观读锁
            stamp = stampedLock.readLock();
            try {
                curX = x;
                curY = y;
            } finally {
                stampedLock.unlockRead(stamp);
            }
        }
        return Math.sqrt(curX * curX + curY * curY); // 大多数情况直接到这里
    }
}
```

**坑 1：以为原子类的复合操作也是原子的**

**❌ 错误代码：**

```java
AtomicInteger stock = new AtomicInteger(10);

    if (stock.get() > 0) {          // 步骤1：检查库存
        stock.decrementAndGet();    // 步骤2：扣减库存
        return true;
    }
    return false;

// 问题：步骤1和步骤2之间，其他线程可能修改了 stock
// 结果：stock 可能变成负数（超卖！）
```

**✅ 正确做法：用 CAS 循环保证原子性**

```java
public boolean buyOne() {
    while (true) {
        int current = stock.get();
        if (current <= 0) {
            return false; // 无货
        }
        // CAS：只有 stock 还是 current 时才扣减
        if (stock.compareAndSet(current, current - 1)) {
            return true; // 扣减成功
        }
        // CAS 失败说明被别人改了，重新读取并重试（自旋）
    }
}
```

### 8.3 避免共享踩坑（6个）

**坑 1：ThreadLocal 没有 remove() 导致内存泄漏（最常见）**

**坑 3：不可变对象中包含可变引用**

```java
    private final List<Item>     items;

        this.items = List.copyOf(items); // 防御性复制，外部修改原 list 不影响
    }

    public List<Item> getItems() {
        return Collections.unmodifiableList(items); // 返回不可修改的视图
        // 或 return List.copyOf(items); // 返回完整副本（开销大但更安全）
    }
```

**坑 5：parallel Stream 中使用非线程安全的共享状态**

### 8.5 锁踩坑（8个）

**坑 1：忘记 unlock() 导致死锁（最常见）**

```java

// ❌ 中间抛异常，unlock 没有执行
lock.lock();
doSomething(); // ← 如果这里抛异常
lock.unlock(); // ← 永远不执行，锁永远不释放，其他线程永远等待

// ✅ 必须用 try-finally
lock.lock();
try {
    doSomething();
} finally {
    lock.unlock(); // 无论如何都执行
}
```

**坑 2：死锁（两个线程互相等对方的锁）**

```java
Object lockA = new Object();
Object lockB = new Object();

// 线程1：先锁A，再锁B
new Thread(() -> {
    synchronized (lockA) {
        sleep(100); // 模拟处理
        synchronized (lockB) { doWork(); } // 等 lockB，但 lockB 被线程2持有
    }
}).start();

// 线程2：先锁B，再锁A
new Thread(() -> {
    synchronized (lockB) {
        sleep(100);
        synchronized (lockA) { doWork(); } // 等 lockA，但 lockA 被线程1持有
    }
}).start();
// 死锁！两个线程互相等待

// ✅ 解决方案：所有线程按相同顺序获取锁
// 规定：永远按 lockA → lockB 的顺序
new Thread(() -> {
    synchronized (lockA) { synchronized (lockB) { doWork(); } }
}).start();
new Thread(() -> {
    synchronized (lockA) { synchronized (lockB) { doWork(); } } // 顺序一致，不会死锁
}).start();
```

**坑 3：synchronized 锁的是对象实例，不是变量**

```java
public class Counter {
    private int count = 0;

    // ❌ 两个方法锁的是不同的对象！
    public void increment() {
        synchronized (new Object()) { // 每次创建新对象，没有互斥效果！
            count++;
        }
    }

    // ❌ 同一实例的两个方法，分别锁 this 和锁 lock，不互斥
    private Object lock = new Object();
    public synchronized void methodA() { count++; }     // 锁 this
    public void methodB() {
        synchronized (lock) { count++; } // 锁 lock，与 methodA 不互斥！
    }

    // ✅ 确保保护同一资源的所有方法用同一把锁
    public synchronized void safeIncrement() { count++; }
    public synchronized int safeGet() { return count; }
}
```

**坑 5：double-checked locking 的单例模式缺少 volatile**

```java
// ❌ 没有 volatile 的 DCL，指令重排可能返回未初始化的对象

    private static Singleton instance; // 缺少 volatile！

    public static Singleton getInstance() {
        if (instance == null) {
            synchronized (Singleton.class) {
                if (instance == null) {
                    instance = new Singleton(); // 这行实际分3步：分配内存、初始化、赋值引用
                    // JVM 可能重排为：分配内存、赋值引用、初始化
                    // 另一个线程可能拿到已赋值但未初始化的引用！
                }
            }
        }
        return instance;
    }
}

// ✅ instance 加 volatile，禁止指令重排
public class SafeSingleton {
    private static volatile SafeSingleton instance; // 加 volatile

    public static SafeSingleton getInstance() {
        if (instance == null) {
            synchronized (SafeSingleton.class) {
                if (instance == null) {
                    instance = new SafeSingleton(); // volatile 保证赋值前初始化已完成
                }
            }
        }
        return instance;
    }
}
```

## 九、企业级最佳实践

### 9.1 完整的企业级并发代码示例

```java
/**
 * 企业级示例：高并发商品库存扣减服务
 * 技术点：
 * - AtomicInteger 做乐观锁
 * - ConcurrentHashMap 做商品缓存
 * - Semaphore 限制并发量
 * - LongAdder 统计成功/失败次数
 * - ReadWriteLock 保护配置
 */
@Service
public class InventoryService {

    // 商品库存（用 AtomicInteger 保证原子操作）
    private final ConcurrentHashMap<Long, AtomicInteger> stockMap =
        new ConcurrentHashMap<>();

    // 限流：最多 100 个并发扣减请求
    private final Semaphore concurrencyLimit = new Semaphore(100);

    // 统计
    private final LongAdder successCount = new LongAdder();
    private final LongAdder failedCount  = new LongAdder();

    // 系统配置（读多写少，用读写锁）
    private final ReadWriteLock configLock    = new ReentrantReadWriteLock();
    private volatile boolean    deductEnabled = true; // volatile 保证可见性

    /**
     * 初始化库存
     */
    public void initStock(Long productId, int quantity) {
        stockMap.put(productId, new AtomicInteger(quantity));
    }

    /**
     * 扣减库存（核心方法）
     */
    public DeductResult deductStock(Long productId, int quantity) {
        // 1. 检查服务是否开启（读配置）
        configLock.readLock().lock();
        try {
            if (!deductEnabled) {
                return DeductResult.disabled();
            }
        } finally {
            configLock.readLock().unlock();
        }

        // 2. 限流：最多 100 并发
        boolean acquired;
        try {
            acquired = concurrencyLimit.tryAcquire(500, TimeUnit.MILLISECONDS);
        } catch (InterruptedException e) {
            Thread.currentThread().interrupt();
            return DeductResult.failed("服务中断");
        }
        if (!acquired) {
            failedCount.increment();
            return DeductResult.rejected("系统繁忙，请稍后重试");
        }

        try {
            // 3. 获取商品库存
            AtomicInteger stock = stockMap.get(productId);
            if (stock == null) {
                failedCount.increment();
                return DeductResult.failed("商品不存在");
            }

            // 4. CAS 循环扣减库存（乐观锁，不加 synchronized）
            int maxRetries = 3;
            for (int retry = 0; retry < maxRetries; retry++) {
                int current = stock.get();

                if (current < quantity) {
                    failedCount.increment();
                    return DeductResult.outOfStock(current);
                }

                // CAS：只有库存还是 current 时，才扣减
                if (stock.compareAndSet(current, current - quantity)) {
                    successCount.increment();
                    return DeductResult.success(current - quantity);
                }
                // CAS 失败：被其他线程修改了，重试
            }

            failedCount.increment();
            return DeductResult.failed("扣减失败，请重试");

        } finally {
            concurrencyLimit.release(); // 必须释放！
        }
    }

    /**
     * 动态关闭/开启扣减功能（写配置）
     */
    public void setDeductEnabled(boolean enabled) {
        configLock.writeLock().lock();
        try {
            this.deductEnabled = enabled;
            log.info("库存扣减功能已{}", enabled ? "开启" : "关闭");
        } finally {
            configLock.writeLock().unlock();
        }
    }

    /**
     * 查询统计数据（只读，线程安全）
     */
    public InventoryStats getStats() {
        return InventoryStats.builder()
            .successCount(successCount.sum())
            .failedCount(failedCount.sum())
            .activeRequests(100 - concurrencyLimit.availablePermits())
            .build();
    }
}
```

### 9.2 企业级代码规范检查清单

```
并发代码 Code Review 检查项：

原子类
  □ 高并发累加用 LongAdder，不用 AtomicLong
  □ 复合操作（检查+修改）使用 CAS 循环，不是分开的 get/set
  □ volatile 不用于 ++ 等非原子操作

集合类
  □ 没有使用 HashMap/ArrayList 在多线程场景下
  □ 没有使用 Executors.newFixedThreadPool（无界队列 OOM 风险）
  □ ConcurrentHashMap 没有传入 null 键或值
  □ 队列的 offer() 返回值有处理（失败不静默丢弃）

ThreadLocal
  □ 所有 set() 都有配套的 remove()（在 finally 块中）
  □ 没有在子线程/线程池中依赖父线程的 ThreadLocal 值

锁
  □ 所有 lock() 都有配套的 unlock()（在 finally 块中）
  □ 没有嵌套锁，或嵌套锁的顺序一致（防死锁）
  □ synchronized 的锁对象是共享的 final 字段，不是局部变量
  □ DCL 单例的 instance 字段有 volatile

工具类
  □ CountDownLatch/CyclicBarrier 的 await() 有超时
  □ Semaphore 的 release() 在 finally 块中
  □ 没有 CountDownLatch 多次 countDown() 的风险

通用
  □ InterruptedException 被正确处理（恢复中断标志）
  □ 共享可变状态最小化（优先不可变对象和局部变量）
  □ 关键并发类有单元测试（多线程并发场景）
```
