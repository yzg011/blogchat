### 1-1-4 CONCAT()

**定义和用法**

将两个或多个表达式加在一起

**参数值**

| 参数     | 描述                     |

### 1-1-5 CONCAT_WS()

**定义和用法**

使用分隔将两个或多个表达式添加在一起

**语法**

CONCAT_WS(*separator*, *expression1*, *expression2*, *expression3*,...)

**参数值**

| 参数                                      | 描述                                                         |
| :---------------------------------------- | :----------------------------------------------------------- |
| separator                                 | 必需。在每个表达式之间添加的分隔符。 如果 separator 为 NULL，则此函数返回 NULL |
| 表达式1<br />表达式2<br />表达式3<br />等 | 必须。要加在一起的表达式。 将跳过具有 NULL 值的表达式        |

```sql
【示例】
  SELECT CONCAT_WS("-", "HELLO", "WORLD", "!") FROM DUAL;
  SELECT CONCAT_WS(NULL, "HELLO", "WORLD", "!") FROM DUAL;
【返回】
  "HELLO-WORLD-!"
  null
```

### 1-1-6 FIELD()

**定义和用法**

FIELD() 函数返回值在值列表中的索引位置。

此函数执行不区分大小写的搜索。

**注释：**如果在值列表中找不到指定的值，此函数将返回 0。如果值为 NULL，此函数将返回 0。

**语法**

FIELD(*value*, *val1*, *val2*, *val3*, ...)

**参数值**

| 参数                         | 描述                     |
| :--------------------------- | :----------------------- |
| *value*                      | 必需。要在列表中搜索的值 |
| *val1*, *val2*, *val3*, .... | 必需。要搜索的值列表     |

```sql
【示例】
  SELECT FIELD("q", "s22", "q1", "l3") from dual;
  SELECT FIELD("q", "s22", "q", "l3") from dual;
【返回】
  0
  2
```

### 1-1-7 FIND_IN_SET()

**定义和用法**

FIND_IN_SET() 函数返回字符串在字符串列表中的位置。

**语法**

FIND_IN_SET(*string*, *string_list*)

**参数值**

| 参数          | 描述                                   |
| :------------ | :------------------------------------- |
| *string*      | 必需。 要搜索的字符串                  |
| *string_list* | 必须。要搜索的字符串值列表（逗号分隔） |

**返回值**

- 如果在*string_list*中没有找到*string*，这个函数返回0
- 如果 *string* 或 *string_list* 为 NULL，则此函数返回 NULL
- 如果*string_list*是一个空字符串（""），这个函数返回0

```sql
【示例】
  SELECT FIND_IN_SET("q", "s,2\q,l");
  SELECT FIND_IN_SET("q", "s,q,l");
  SELECT FIND_IN_SET(null, "s,2\q,l");
  SELECT FIND_IN_SET("q", null);
  SELECT FIND_IN_SET("q", "");
【返回】
  0
  2
  null
  0
```

### 1-1-8 FORMAT()

**定义和用法**

FORMAT() 函数将数字格式化为像"#,###,###.##"这样的格式，四舍五入到指定的小数位数，然后将结果作为字符串返回。

**语法**

FORMAT(*number*, *decimal_places*)

**参数值**

| 参数             | 描述                                                         |
| :--------------- | :----------------------------------------------------------- |
| *number*         | 必需。要格式化的数字                                         |
| *decimal_places* | 必需。*number的小数位数。*如果该参数为0，则该函数返回一个没有小数位的字符串 |

```sql
【示例】
  SELECT FORMAT(250500.5634, 2) from dual;
  SELECT FORMAT(250500.5634, 0) from dual;
【返回】
  "250,500.56"
  "250,501"
```

### 1-1-9 INSERT()

**定义和用法**

INSERT() 函数在字符串中的指定位置插入一个字符串，并插入一定数量的字符。

**语法**

INSERT(*string*, *position*, *number*, *string2*)

**参数值**

的位置 中的字符串

| 参数       | 描述                   |
| :--------- | :--------------------- |
| *string*   | 必需。将要修改的字符串 |
| *position* | 必需。插入*string2*    |
| *number*   | 必需。要替换的字符数   |
| *string2*  | 必需。要插入到*string* |

**返回值**

- 如果 *position* 超出 *string* 的长度，这个函数返回 *string*
- 如果 *number* 大于 *string* 其余部分的长度，此函数将 *string* 替换为 *position* 直到 *string*

```sql
【示例】
  SELECT INSERT("Hello World", 1, 5, "World") from dual;
  
  将字符串"no"插入字符串"Hello World"。 替换三个字符，从位置 11 开始
  SELECT INSERT("Hello World", 11, 3, "no") from dual;
【返回】
  "World World"
  
  "Hello Worlno"
  
```

### 1-1-14 LOCATE()

**定义和用法**

此函数执行不区分大小写的搜索。

**语法**

LOCATE(*substring*, *string*, *start*)

**参数值**

中搜索的子字符串

| 参数        | 描述                                    |
| :---------- | :-------------------------------------- |
| *substring* | 必需。要在 *string*                     |

```sql
【示例】
  在字符串"Hello World"中搜索"wo"（从位置3开始），返回位置：
  SELECT LOCATE("wo", "Hello World", 8) from dual;
  在字符串"Hello World"中搜索"wo"，返回位置：
  SELECT LOCATE("wo", "Hello World") from dual;

  0
  7
```

的天数

### CASE

**定义和用法**

CASE 语句遍历条件并在满足第一个条件时返回一个值（如 IF-THEN-ELSE 语句）。 因此，一旦条件为真，它将停止读取并返回结果。

如果没有条件为真，它将返回 ELSE 子句中的值。

如果没有ELSE部分且没有条件为真，则返回NULL。

**语法**

CASE
  WHEN *condition1* THEN *result1*
  WHEN *condition2* THEN *result2*
  WHEN *conditionN* THEN *resultN*
  ELSE *result*
END;

**参数值**

| 参数                                    | 描述                                        |
| :-------------------------------------- | :------------------------------------------ |
| *condition1, condition2, ...conditionN* | 必需。条件。 它们的评估顺序与列出的顺序相同 |
| *result1, result2, ...resultN*          | 必需。条件为真时返回的值                    |

```sql
【示例】
SELECT CASE
           WHEN
               1 > 30 THEN
               '1>30'
           WHEN 30 = 30 THEN
               '30'
           ELSE 'else'
           END
FROM DUAL;
【返回】	
  30
```

## 索引类型

MySQL 索引是提升查询效率的核心机制，其类型可从 **数据结构、逻辑功能、物理存储** 三个核心维度划分，以下结合开发实战场景（如你之前提到的“知识点服务”的 `knowledge_point` 表），详细讲解各类索引的特点、适用场景及最佳实践：

### 一、核心维度1：按「数据结构」划分（底层实现）
这是索引最本质的分类，决定了索引的查询效率、适用场景，MySQL 5.7+ 主流支持以下4种：

| 索引类型       | 核心特点                                                                 | 适用场景                                  | 优缺点                                  | 示例（知识点服务）|
|----------------|--------------------------------------------------------------------------|-------------------------------------------|-----------------------------------------|-----------------------------------------|
| B+Tree 索引    | 1. 所有数据存在叶子节点，按顺序排列；<br>2. 支持范围查询、排序、分组；<br>3. InnoDB/MyISAM 默认索引结构 | 绝大多数场景（等值查询、范围查询、排序）  | 优点：通用、效率高；<br>缺点：写入略慢   | 知识点ID、分类ID、创建时间等字段的索引 |
| Hash 索引      | 1. 基于哈希表实现，通过哈希值定位数据；<br>2. 仅支持等值查询（=、IN），不支持范围/排序 | 高频等值查询场景（如缓存表、字典表）      | 优点：等值查询极快；<br>缺点：不支持范围、易哈希冲突 | 知识点唯一编码（`unique_code`）的索引   |
| Full-Text 索引 | 1. 全文索引，针对文本内容（char/varchar/text）；<br>2. 支持关键词匹配（MATCH AGAINST） | 文本检索场景（如知识点标题、内容搜索）    | 优点：支持模糊文本检索；<br>缺点：仅MyISAM（5.6前）/InnoDB（5.6后）支持，精度有限 | 知识点标题（`title`）、内容（`content`） |
| R-Tree 索引    | 1. 基于空间数据结构，用于地理空间数据；<br>2. 支持空间范围查询（如距离、包含） | 地理空间场景（如知识点关联的地理位置）    | 优点：空间查询高效；<br>缺点：适用场景极窄 | 知识点关联的经纬度（`location`）字段    |

> 重点：**B+Tree 是MySQL最核心、最常用的索引类型**，后文逻辑功能类索引均基于B+Tree实现（除全文/哈希外）。

### 二、核心维度2：按「逻辑功能」划分（开发中显式创建）
这是开发人员最常接触的分类，对应 `CREATE INDEX` 语句的不同用法：

#### 1. 主键索引（Primary Key）
- **定义**：唯一标识一行数据的索引，一张表仅能有1个，默认非空（NOT NULL）。
- **特点**：InnoDB 中主键索引是「聚簇索引」（叶子节点存整行数据），性能最优。
- **创建示例**（知识点表）：
  ```sql
  CREATE TABLE `knowledge_point` (
    `id` BIGINT NOT NULL AUTO_INCREMENT COMMENT '知识点ID',
    `title` VARCHAR(200) NOT NULL COMMENT '知识点标题',
    `category_id` BIGINT NOT NULL COMMENT '分类ID',
    PRIMARY KEY (`id`) -- 主键索引
  ) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
  ```

#### 2. 唯一索引（Unique Index）
- **定义**：保证索引列的值唯一（允许NULL，NULL可重复），一张表可多个。
- **适用场景**：唯一标识字段（如知识点编码、外部关联ID）。
- **创建示例**：
  ```sql
  -- 知识点唯一编码的唯一索引
  CREATE UNIQUE INDEX idx_kp_unique_code ON knowledge_point(unique_code);
  ```

#### 3. 普通索引（Normal Index）
- **定义**：最基础的索引，无唯一性约束，仅用于加速查询。
- **适用场景**：非唯一、高频查询的字段（如分类ID、创建人ID）。
- **创建示例**：
  ```sql
  -- 知识点分类ID的普通索引
  CREATE INDEX idx_kp_category_id ON knowledge_point(category_id);
  ```

#### 4. 组合索引（Composite Index）
- **定义**：基于多个字段的联合索引，遵循「最左匹配原则」（查询条件需从左到右匹配索引字段）。
- **核心坑点**：索引字段顺序决定查询效率（高频查询字段放左侧）。
- **适用场景**：多字段联合查询（如按“分类ID+创建时间”查询知识点）。
- **创建示例**：
  ```sql
  -- 组合索引：分类ID（高频）+ 创建时间（范围）
  CREATE INDEX idx_kp_category_create_time ON knowledge_point(category_id, create_time);
  -- 有效查询：WHERE category_id=1 AND create_time > '2025-01-01'（匹配最左）
  -- 无效查询：WHERE create_time > '2025-01-01'（跳过左侧字段，索引失效）
  ```

#### 5. 前缀索引（Prefix Index）
- **定义**：针对长文本字段，仅索引字段的前N个字符，减少索引存储空间。
- **适用场景**：长字符串字段（如知识点标题、内容摘要）。
- **创建示例**：
  ```sql
  -- 知识点标题取前50个字符做前缀索引
  CREATE INDEX idx_kp_title_prefix ON knowledge_point(title(50));
  ```

#### 6. 空间索引（Spatial Index）
- **定义**：基于R-Tree实现，仅支持空间数据类型（GEOMETRY、POINT等）。
- **适用场景**：地理空间相关的知识点（如线下课程知识点的位置）。
- **创建示例**：
  ```sql
  ALTER TABLE knowledge_point ADD SPATIAL INDEX idx_kp_location (location);
  ```

### 三、核心维度3：按「物理存储」划分（InnoDB 特有）
仅针对 InnoDB 引擎（MySQL 主流引擎），决定了索引与数据的存储关系：

| 索引类型       | 核心特点                                                                 | 对应场景                |
|----------------|--------------------------------------------------------------------------|-------------------------|
| 聚簇索引（Clustered） | 1. 索引与数据存储在一起，叶子节点存整行数据；<br>2. InnoDB 中主键索引就是聚簇索引；<br>3. 一张表仅1个 | 主键查询（效率最高）    |
| 非聚簇索引（Secondary） | 1. 索引与数据分离，叶子节点存「主键值」；<br>2. 所有非主键索引（唯一、普通、组合等）均为非聚簇索引；<br>3. 查询需「回表」（通过主键查聚簇索引） | 非主键字段查询          |

> 关键优化点：避免频繁回表！
> 例如：查询 `SELECT title FROM knowledge_point WHERE category_id=1`，若仅创建 `idx_kp_category_id`（非聚簇索引），需先查索引得主键ID，再查聚簇索引取title（回表）；
> 优化方案：创建「覆盖索引」`CREATE INDEX idx_kp_category_title ON knowledge_point(category_id, title)`，索引包含title，无需回表。

```sql
　　soundex('two'),soundex('too'),soundex('to')，他们的结果都是T000
```

```sql
语法 FUNCTION_NAME(<参数>,…) OVER (<PARTITION BY 表达式,…> <ORDER BY 表达式 <ASC DESC> <NULLS FIRST NULLS LAST>> <WINDOWING子句>) PARTITION子句 ORDER BY子句 WINDOWING子句 缺省时相当于RANGE UNBOUNDED PRECEDING 
1. 值域窗(RANGE WINDOW) 
RANGE N PRECEDING 仅对数值或日期类型有效,选定窗为排序后当前行之前,某列(即排序列)值大于/小于(当前行该列值 –/+ N)的所有行,因此与ORDER BY子句有关系。 
2. 行窗(ROW WINDOW) 
ROWS N PRECEDING 选定窗为当前行及之前N行。 
还可以加上BETWEEN AND 形式,例如RANGE BETWEEN m PRECEDING AND n FOLLOWING 
函数 AVG(<distinct all> eXPr) 
一组或选定窗中表达式的平均值 CORR(expr, expr) 即COVAR_POP(exp1,exp2) / (STDDEV_POP(expr1) * STDDEV_POP(expr2)),两个表达式的互相关,-1(反相关) ~ 1(正相关),0表示不相关
```

#### 2-7-1 DUMP(w[,x[,y[,z]]])

```sql
DUMP(w[,x[,y[,z]]])
【功能】返回数据类型、字节长度和在内部的存储位置.
【参数】
 w为各种类型的字符串（如字符型、数值型、日期型……）
 x为返回位置用什么方式表达，可为:8,10,16或17，分别表示：8/10/16进制和字符型，默认为10。
 y和z决定了内部参数位置


【返回】类型 <[长度]>，符号/指数位 [数字1，数字2，数字3，......，数字20]
如：Typ=2 Len=7: 60,89,67,45,23,11,102

SELECT DUMP('ABC',1016) FROM dual;　 
返回结果为：Typ=96 Len=3 CharacterSet=ZHS16GBK: 41,42,43 

　　代码 数据类型
　　0 对应 VARCHAR2
　　1 对应 NUMBER
　　8 对应 LONG
　　12 对应 DATE
　　23 对应 RAW
　　24 对应 LONG RAW
　　69 对应 ROWID
　　96 对应 CHAR
　　106 对应 MSSLABEL



各位的含义如下:

1.类型: Number型,Type=2 (类型代码可以从Oracle的文档上查到)
2.长度:指存储的字节数
3.符号/指数位
在存储上,Oracle对正数和负数分别进行存储转换:
正数：加1存储(为了避免Null)
负数：被101减,如果总长度小于21个字节，最后加一个102(是为了排序的需要)

指数位换算:
正数：指数=符号/指数位 - 193 (最高位为1是代表正数) 
负数：指数=62 - 第一字节

4.从<数字1>开始是有效的数据位

从<数字1>开始是最高有效位,所存储的数值计算方法为：

将下面计算的结果加起来：

每个<数字位>乘以100^(指数-N) (N是有效位数的顺序位，第一个有效位的N=0)

5、举例说明

SQL> select dump(123456.789) from dual;
返回：Typ=2 Len=6: 195,13,35,57,79,91
 
<指数>:    195 - 193 = 2 
<数字1>     13 - 1     = 12 *100^(2-0) 120000 
<数字2>     35 - 1     = 34 *100^(2-1) 3400 
<数字3>     57 - 1     = 56 *100^(2-2) 56 
<数字4>     79 - 1     = 78 *100^(2-3) .78 
<数字5>     91 - 1     = 90 *100^(2-4) .009 
　                            123456.789 

SQL> select dump(-123456.789) from dual;
返回：Typ=2 Len=7: 60,89,67,45,23,11,102
算法：
<指数>  62 - 60 = 2(最高位是0，代表为负数) 
<数字1> 101 - 89 = 12 *100^(2-0) 120000 
<数字2> 101 - 67 = 34 *100^(2-1) 3400 
<数字3> 101 - 45 = 56 *100^(2-2) 56 
<数字4> 101 - 23 = 78 *100^(2-3) .78 
<数字5> 101 - 11 = 90 *100^(2-4) .009 
　                              123456.789(-) 

现在再考虑一下为什么在最后加102是为了排序的需要，-123456.789在数据库中实际存储为

60,89,67,45,23,11 

而-123456.78901在数据库中实际存储为 

60,89,67,45,23,11,91 

可见，如果不在最后加上102，在排序时会出现-123456.789<-123456.78901的情况。
```

```sql

when <表达式条件值1> then <满足条件时返回值1> 
[when <表达式条件值2> then <满足条件时返回值2>

【功能】当：<表达式>＝<表达式条件值1……n> 时，返回对应 <满足条件时返回值1……n>
```

```sql
各种操作符的运算优先级
\转义符
(), (?:), (?=), [] 圆括号和方括号
*, +, ?, {n}, {n,}, {n,m} 限定符
^, $, anymetacharacter 位置和顺序
```

范例：

**regexp_like**

--查询value中以1开头60结束的记录并且长度是7位

```sql
select * from fzq where value like '1____60';
select * from fzq where regexp_like(value,'1....60');
```

--查询value中以1开头60结束的记录并且长度是7位并且全部是数字的记录。 --使用like就不是很好实现了。

```sql
select * from fzq where regexp_like(value,'1[0-9]{4}60');
```

-- 也可以这样实现，使用字符集。

```sql
select * from fzq where regexp_like(value,'1[[:digit:]]{4}60');
```

-- 查询value中不是纯数字的记录

```sql
select * from fzq where not regexp_like(value,'^[[:digit:]]+$');
```

-- 查询value中不包含任何数字的记录。

```sql
select * from fzq where regexp_like(value,'^[^[:digit:]]+$');
```

--查询以12或者1b开头的记录.不区分大小写。

```sql
select * from fzq where regexp_like(value,'^1[2b]','i');
```

--查询以12或者1b开头的记录.区分大小写。

```sql
select * from fzq where regexp_like(value,'^1[2B]');
```

-- 查询数据中包含空白的记录。

```sql
select * from fzq where regexp_like(value,'[[:space:]]');
```

--查询所有包含小写字母或者数字的记录。

```sql
select * from fzq where regexp_like(value,'^([a-z]+|[0-9]+)$');
```

--查询任何包含标点符号的记录。

```sql
select * from fzq where regexp_like(value,'[[:punct:]]');
```

**相关函数**

> 函数中pattern为正则表达式，最多可以包含512个字节。

**REGEXP_SUBSTR**

REGEXP_SUBSTR函数使用正则表达式来指定返回串的起点和终点。 语法：

```sql
regexp_substr(source_string,pattern[,position[,occurrence[,match_parameter]]])
```

source_string：源串，可以是常量，也可以是某个值类型为串的列。

position：从源串开始搜索的位置。默认为1。

occurrence：指定源串中的第几次出现。默认值1.

match_parameter：文本量，进一步订制搜索，取值如下：

> 

- 'i'     用于不区分大小写的匹配。
- 'c'    用于区分大小写的匹配。
- 'n'    允许将句点“.”作为通配符来匹配换行符。如果省略改参数，句点将不匹配换行符。
- 'm'   将源串视为多行。即将“^”和“$”分别看做源串中任意位置任意行的开始和结束，而不是看作整个源串的开始或结束。如果省略该参数，源串将被看作一行来处理。
- 如果取值不属于上述中的某个，将会报错。如果指定了多个互相矛盾的值，将使用最后一个值。如'ic'会被当做'c'处理。
- 省略该参数时：默认区分大小写、句点不匹配换行符、源串被看作一行。

**REGEXP_INSTR**

REGEXP_INSTR函数使用正则表达式返回搜索模式的起点和终点（整数）。如果没有发现匹配的值，将返回0。 语法：

```sql
regexp_instr(source_string,pattern[,position[,occurrence[,return_option[,match_parameter]]]])
```

> REGEXP_INSTR函数常常会被用到where子句中。

**REGEXP_LIKE**

通常使用REGEXP_LIKE进行模糊匹配。 语法：

```sql
regexp_like(source_string,pattern[match_parameter])
```

> 该函数可以使用前面介绍的所有搜索功能作为REGEXP_LIKE搜索的一部分，可以是非常复杂的搜索变得简单。

**REPLACE和REGEXP_REPLACE**

REPLACE函数用于替换串中的某个值。 语法：

```sql
replace(char,search_string[,replace_string])
```

> 如果不指定replace_string，会将搜索到的值删除。

REGEXP_REPLACE是REPLACE的增强版，支持正则表达式，扩展了一些功能。 语法：

```sql
regexp_replace(source_string,pattern[,replace_string[,position[,occurrence[,match_parameter]]]])
```

> replace_string表示用什么来替换source_string中与pattern匹配的部分。 occurrence为非负整数，0表示所有匹配项都被替换，为正数时替换第n次匹配。

**REGEXP_COUNT**

REGEXP_COUNT函数返回在源串中出现的模式的次数，作为对REGEXP_INSTR函数的补充。 虽然COUNT是一个集合函数，操作的是行组，但是REGEXP_COUNT是单行函数，分别计算每一行。 语法：

```sql
regexp_count(source_char,pattern[,position[,match_param]])
```

> REGEXP_COUNT返回pattern在source_char串中出现的次数。如果未找到匹配，函数返回0。 metch_param参数，相对于前面介绍的match_parameter参数多一个取值“x”。 'x'：忽略空格字符。默认情况下，空格与自身想匹配。 metch_param如果指定了多个互相矛盾的值，将使用最后一个值。

# 执行计划
|                |                                                                                                                                |
| -------------- | ------------------------------------------------------------------------------------------------------------------------------ |
| 字段名            | 描述                                                                                                                             |
| id             | select序列号，有几个select结果集就有几个id，id的顺序是按照select出现的顺序增长的，id越大越先执行，id相同则从上到下执行，id为NULL最后执行                                           |
| select_type    | 表示对应行是简单查询还是复杂查询                                                                                                               |
| table          | 表示对应行要访问的哪张表                                                                                                                   |
| partitions     | 表示分区表命中的分区情况，非分区表该字段为空（null）                                                                                                   |
| type           | 关联类型或访问类型，表示要决定如何查找表中的行，查找数据行记录的大概范围，从高到低依次 system > const > eq_ref > ref > range > index > all ; 一般来说要保证查询达到range级别，最好达到ref级别 |
| popssible_keys | 表示执行当前行对应语句时可能使用到的索引                                                                                                           |
| keys           | 表示执行当前行对应语句时实际使用到的索引                                                                                                           |
| key_len        | 表示执行当前行对应语句时实际使用到的索引的长度                                                                                                        |
| ref            | 当使用索引列等值查询时，与索引列等值匹配的对象信息                                                                                                      |

| filtered       | 使用explain extended时会出现这个列，5.7之后的版本默认有这个字段，不需要使用explain extended了。该字段表示存储引擎返回的数据在server层过滤后，剩下多少满足查询的记录数量的比例，注意是百分比，不是具体记录数     |
| Extra          | 额外信息                                                                                                                           |
## select_type
1. SIMPLE：不使用UNION和子查询的简单查询
2. PRIMARY：最外层的select查询
3. UNION：union关键字后面一个select查询，不依赖外部的查询结果集
4. UNION RESULT：union结果集
5. SUBQUERY：子查询的第一个select查询，不依赖外部的查询结果集
6. DERIVED：用于from子句中有子查询的情况，MySQL会递归这些子查询，把结果放到临时表中

## table
MySQL规定Explain结果集中的每条记录对应某个单表的访问方法，table字段表示该单表的表名，可能为真实表名或别名或衍生表名

## type
从高到低依次为 system > const > eq_ref > ref > range > index > all，一般要保证查询达到range级别，最好达到ref级别  
1）system：对表中只有一条记录的表进行查询并且表的数据统计是精确时就会出现这种type  
2）const：对主键(primary key)或者唯一性二级(unique key)索引进行等值匹配时，如果主键或者二级索引是由多个字段构成，那么参与构建索引的字段都要进行等值匹配，才会出现const  
3）eq_ref：主键(primary key)或者唯一性二级(unique key)索引的所有部分被连接使用，最多只会返回一条符合条件的记录，这是在 const 外最好的联接类型  
4）ref：当使用普通的二级索引进行等值匹配时，就会出现ref  
5）range：范围扫描通常出现在in(),between,>,<,>=等操作中，使用一个索引来检索给定范围的行
6）index：全索引扫描，一般是扫描某个二级索引，这种扫描不会从索引树根节点开始快速查找，而是直接对二级索引的叶子节点遍历和扫描，速度还是比较慢的，这种查询一般为使用覆盖索引，二级索引一般比较小，所以这种通常比ALL快一些  
7）ALL：全表扫描，扫描聚簇索引的所有叶子节点。通常情况下这需要增加索引来进行优化  

## possible_keys
表示执行当前行对应语句时可能使用到的索引

## key
表示执行当前行对应语句实际要使用的索引，若未使用索引则为NULL

## key_len
表示执行当前行对应语句实际使用索引的字节数，通过该值可推测出具体使用索引的哪些列  
key_len计算规则如下：  
1.字符串，char(n)和varchar(n)，5.0.3之后版本，n均代表字符数，而不是字节数，如果是utf-8，一个数字或字母占1个字节，一个汉字占3个字节  
○ char(n)：如果存汉字长度就是 3n 字节  
○ varchar(n)：如果存汉字则长度是 3n + 2 字节，加的2字节用来存储字符串长度，因为 varchar是变长字符串  
2.数值类型  
○ tinyint：1字节  
○ smallint：2字节  
○ int：4字节  
○ bigint：8字节  
3.时间类型  
○ date：3字节  
○ timestamp：4字节  
○ datetime：8字节  
4.如果字段允许为 NULL，需要1字节记录是否为 NULL  
索引最大长度是768字节，当字符串过长时，mysql会做一个类似左前缀索引的处理，将前半部分的字符提取出来做索引。  

## ref
表示在实际要使用的索引中，表查找值所用到的列或常量，常见的有：const（常量），字段名（如user.id）

## rows
表示mysql估计要读取并检测的行数，并非最终结果集行数

## Extra
1）Using index：使用覆盖索引
覆盖索引定义：mysql执行计划explain结果里的key有使用索引，如果select后面查询的字段都可以从这个索引树获取，这种情况一般是用到了覆盖索引，extra里一般都有using index；覆盖索引一般针对的是辅助索引，整个查询结果只通过辅助索引就能拿到结果，不需要通过辅助索引树找到主键，再通过主键去主键索引树里获取其它字段值  

2）Using where：使用 where 语句来处理结果，并且查询的列未被索引覆盖
3）Using index condition：查询的列不完全被索引覆盖，where条件中是一个前导列的范围  

4）Using temporary：mysql需要创建一张临时表来处理查询。出现这种情况一般是要进行优化的，首先是想到用索引来优化（没有索引，此时创建了张临时表来distinct；user_role.uid建了索引所以直接使用索引来distinct，此时extra为using index）
5）Using filesort：将用外部排序而不是索引排序，数据较小时从内存排序，否则需要在磁盘完成排序。这种情况下一 般也是要考虑使用索引来优化（没有索引，mysql会全表扫描保存排序关键字name和对应id，然后排序name并检索行记录；若name建了索引则再查询时extra为using index）
